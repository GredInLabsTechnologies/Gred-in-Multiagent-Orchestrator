## Detailed Gaps

### 🔴 GAP #1: Missing MCP Dependency (BLOCKER)
**Severity:** CRITICAL
**Component:** Backend Startup
**Error:**
```
ModuleNotFoundError: No module named 'mcp'
```

**Root Cause:** `requirements.txt` missing `fastmcp` and `mcp` packages

**Impact:**
- Backend fails to start out-of-the-box
- Requires manual `pip install fastmcp mcp`
- CI/CD pipelines would fail
- New developers blocked for 10-20 minutes

**Fix:**
```bash
# Add to requirements.txt:
fastmcp>=3.1.0
mcp>=1.26.0
```

**Priority:** P0 (Must fix before next release)

---

### 🔴 GAP #2 & #3: Authentication System Confusion (BLOCKER)
**Severity:** HIGH
**Component:** Security / Auth

**Issue 1: Three tokens with unclear purpose**
- `ORCH_TOKEN` (main token)
- `ORCH_ACTIONS_TOKEN` (60 req/min, "actions" role)
- `ORCH_OPERATOR_TOKEN` (200 req/min, "operator" role)

**Confusion Points:**
- User comment: *"como que 3 tokens? que sentido tiene?"*
- User clarification: *"ya no usamos actions, ahora usamos chatgpt apps, no actions"*
- No clear documentation on when to use each token
- Different permissions per token (operator can't change providers, actions is read-only)

**Issue 2: Token mismatch on startup**
- User creates `.orch_token` → Backend ignores it and generates new token
- Backend loads 3 different tokens from 3 files at startup
- Token in `.orch_token` doesn't match backend's loaded token → 401 Unauthorized
- Requires finding and using `tools/gimo_server/.orch_operator_token` instead

**Impact:**
- 30+ minutes wasted debugging auth failures
- Extremely poor first-run experience
- Violates principle of least surprise

**Fix:**
1. **Short term:** Document token system clearly in README
2. **Long term:** Simplify to single admin token with optional role tokens
3. Deprecate "actions" token if not used for GitHub Actions/GPTs
4. Backend should respect user-provided `.orch_token` on first run

**Priority:** P0

---

### 🔴 GAP #7: Schema Validation Mismatch (BLOCKER)
**Severity:** CRITICAL
**Component:** Plan Generation / LLM Integration

**Error:**
```json
{
  "status": "error",
  "error": "Plan generation failed: 2 validation errors for OpsPlan\ntasks.0.agent_assignee.role\n  Input should be 'orchestrator', 'worker' or 'external_action' [type=literal_error, input_value='Lead Orchestrator', ..."
}
```

**Root Cause:**
- LLM (qwen2.5-coder:3b) generates `role='Lead Orchestrator'`
- Schema only accepts: `'orchestrator'`, `'worker'`, `'external_action'`
- System prompts don't constrain LLM output strictly enough
- Small model (3B params) not capable enough for structured output

**Impact:**
- **100% plan generation failure rate** with local Ollama
- User expectation: *"el orquestador como va a ser ollama? en todo caso deberias ser tu, o un agente claude haiku"*
- Waste of LLM calls (tokens consumed but no valid output)

**Evidence:**
- 3/3 recent drafts: `status: "error"`
- Same validation error repeated across attempts

**Fix:**
1. **Immediate:** Add strict schema examples in system prompt:
   ```
   VALID roles: "orchestrator" | "worker" | "external_action"
   INVALID roles: "Lead Orchestrator", "Orchestrator Agent", etc.

   Example:
   "agent_assignee": {"role": "orchestrator", "model": "..."}
   ```
2. Use JSON schema validation in prompt (if LLM supports)
3. **Default to capable model** (GPT-4, Claude Sonnet/Opus) for plan generation
4. Add fallback: if validation fails, retry with better prompt

**Priority:** P0

### 🔴 GAP #9: Provider Config Doesn't Reload (BLOCKER)
**Severity:** MEDIUM
**Component:** Configuration Management

**Issue:**
- Edited `.orch_data/ops/provider.json`:
  ```json
  "active": "claude-account"  // Changed from "openai"
  ```
- Restarted backend
- Backend still reports:
  ```
  active: "local_ollama"
  ```

**Root Cause:**
- Backend caches provider config in memory
- File changes not detected/reloaded
- Inconsistency between file and runtime state

**Impact:**
- Requires full restart + manual verification
- Config changes don't take effect reliably
- Debugging nightmare (file says X, backend does Y)

**Fix:**
1. Add hot-reload for provider config
2. Or add endpoint: `POST /ops/provider/reload`
3. Log warning if file != runtime state

**Priority:** P1
