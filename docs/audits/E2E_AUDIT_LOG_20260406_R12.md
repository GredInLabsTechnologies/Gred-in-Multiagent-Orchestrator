### Analysis

R12 reveals a **systemic problem with fix persistence**: R11 claimed 11 of 13 issues resolved with 1377 passing tests, yet 7 of those fixes are demonstrably broken in the live system. This points to a structural gap between what tests validate and what the live MCP bridge executes.

**The critical insight**: GIMO's MCP surface — its primary integration with Claude — has **persistent, round-over-round failures** that do not get fixed despite test suites passing. The root cause is likely that:

1. The MCP manifest (`manifest.py`) is generated from OpenAPI but drifts because the generator doesn't run in CI.
2. The MCP bridge (`native_tools.py`, `governance_tools.py`) uses `proxy_to_api()` which depends on correct URL paths in the manifest.
3. Tests exercise HTTP endpoints directly (via `TestClient`), not through the MCP→HTTP proxy chain, so manifest drift is invisible to the test suite.

**The meta-issue (R12-#1)** is the most important finding: the audit→fix→test cycle has a systematic blind spot. Fixes pass tests but don't fix the actual MCP surface. Until the test suite includes MCP-level integration tests that exercise the full `tool_call → proxy_to_api → HTTP endpoint` chain, this pattern will recur.

**R12 vs R11**: Same issue count (13) but different composition. R12 has 5 BLOCKERS vs R11's 4 because the _inability to execute a run through any surface_ (R12-#5) is a new compound blocker. The 2 confirmed R11 fixes (graph/capabilities CLI, agent_id regex) are genuine improvements that survive restart.

**Probe B status**: OpenAI worker execution remains BLOCKED for the third consecutive round. Plan generation works, governance is provider-agnostic at the policy layer, but actual execution requires fixing R12-#5 (run creation) and R12-#13/R11-#13 (Windows subprocess).
