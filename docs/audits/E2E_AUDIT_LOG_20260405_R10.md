## Issues

### [BLOCKER] #1 — AnthropicAdapter builds double `/v1/v1/messages` URL — ALL LLM operations fail

- **Command**: `python gimo.py chat -m "Say hello" -w gimo_prueba`
- **Also fails**: `python gimo.py plan "..." -w gimo_prueba --no-confirm`
- **Expected**: LLM responds normally
- **Actual**: `Error communicating with LLM: Client error '404 Not Found' for url 'https://api.anthropic.com/v1/v1/messages'`
- **Severity**: critical — **ALL LLM functionality broken** (chat, plan, run)
- **Root cause**: `DEFAULT_BASE_URLS["claude"]` = `"https://api.anthropic.com/v1"` (with `/v1`), but `AnthropicAdapter` constructs `{base_url}/v1/messages` internally, producing `https://api.anthropic.com/v1/v1/messages`.
  - `adapter_registry.py:30`: `base_url = entry.base_url or DEFAULT_BASE_URLS.get(canonical_type, "https://api.anthropic.com")` — picks up `/v1` suffix
  - `anthropic_adapter.py:180,212,225`: `f"{self.base_url}/v1/messages"` — adds another `/v1`
  - `anthropic_adapter.py:29`: default `base_url = "https://api.anthropic.com"` (correct, no `/v1`) — but this default is never used because `adapter_registry.py` always passes the `DEFAULT_BASE_URLS` value

### [GAP] #2 — `repos list` still shows `dummy_repo` from pytest temp dir

- **Command**: `python gimo.py repos list`
- **Expected**: Only user-managed repos
- **Actual**: Shows `dummy_repo` from `C:\Users[USER]\AppData\Local\Te...` alongside the real repo. The R6/R7 registry-only fix reduced from 17 to 2 repos, but the pytest temp repo persists (likely registered during test runs).
- **Severity**: low
- **Suspicion**: `dummy_repo` was auto-registered in the repo registry during test execution and never cleaned up.

### [FRICTION] #3 — Thread list shows many "New Conversation" titles from pre-R6 era

- **Command**: `python gimo.py threads list`
- **Expected**: All threads have descriptive titles
- **Actual**: 9 of 20 threads still titled "New Conversation" (created before R6 title fix). Recent threads (post-R9) correctly show auto-generated titles.
- **Severity**: low
- **Note**: Not a regression — these are historical threads. Title generation works for new threads.

### [INCONSISTENCY] #4 — `status` shows `Permissions: suggest` — unclear what this means

- **Command**: `python gimo.py status`
- **Expected**: Permissions field shows something actionable (e.g., "workspace_safe" or "can write files")
- **Actual**: `Permissions: suggest` — not documented, not actionable. User doesn't know if they can write files.
- **Severity**: low
