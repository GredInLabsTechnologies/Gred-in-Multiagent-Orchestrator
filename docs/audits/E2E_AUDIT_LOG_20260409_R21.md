## 8. Findings (full issue list)

### [BLOCKER] R21-001 — `gimo_create_draft` MCP path defaults `operator_class=human_ui`, breaking R20-001 at runtime
- **Probe**: A
- **Tool**: `mcp__gimo__gimo_create_draft(task_instructions="...")`
- **Expected**: per R20 Change 1, MCP-context drafts should persist `operator_class="cognitive_agent"` so the policy gate's `fallback_to_most_restrictive_human_review` whitelist permits autonomous execution.
- **Actual**: response payload `"operator_class": "human_ui"`. Subsequent `auto_run` halts at `Pipeline halted: HUMAN_APPROVAL_REQUIRED`.
- **Evidence**: draft `d_1775702359655_359efa`, run `r_1775702365944_1bf912`.
- **Suspicion**: the MCP bridge tool wrapper (`mcp_bridge/native_tools.py` create_draft path) does NOT inject `operator_class="cognitive_agent"` into the `_draft.create_draft` call's context, even though `BrokerTaskDescriptor` was wired up. R20 conformance test asserted draft contract via direct service call, not via the MCP tool surface.

### [CRITICAL] R21-003 — `chat` path bypasses provider readiness chokepoint
- **Probe**: C
- **Command**: `python gimo.py chat -m "say hi in 3 words"`
- **Expected**: same `PROVIDER_NOT_READY:openai:auth_missing` chokepoint that `spawn_subagent` enforces.
- **Actual**: chat dispatched directly to OpenAI and returned `Client error '401 Unauthorized'` from `api.openai.com`. Trust counter for `model:openai:gpt-4o` incremented to `failures=8`.
- **Evidence**: `model:openai:gpt-4o` GICS entry shows `failures=8, policy=blocked` while still being the active model.
- **Suspicion**: `_AUTH_REQUIRED_PROVIDERS` is enforced only in `SubAgentManager._require_provider_readiness`. The agentic-loop chat path goes through `provider_service`/`openai_compat.chat_with_tools` which has no readiness preflight. Cross-surface chokepoint is not unified.

### [CRITICAL] R21-005 — `gimo ps` cannot detect a running server
- **Probe**: C
- **Command**: `gimo.cmd ps`, `python gimo.py ps`
- **Expected**: detect the live server on 9325 and report pid/uptime.
- **Actual**: both report `No GIMO instances found on port(s): 9325` while `/health` returns 200 and pid 16524 is the live server.
- **Suspicion**: `ps` walks the OS process table for a marker that no longer matches detached/launcher-spawned servers (R19/R20 changed launcher to use Win32 Job Objects with KILL_ON_JOB_CLOSE — the marker may have moved).

### [CRITICAL] R21-006 — `gimo.cmd doctor` (and other subcommands) is broken
- **Probe**: C
- **Command**: `./gimo.cmd doctor`
- **Expected**: same output as `python gimo.py doctor`.
- **Actual**: shell errors `'tlocal' no se reconoce como un comando interno o externo` and similar for `GIMO`, `Usage:`. The .cmd file appears to splat output through `cmd /c` in a way that gets re-interpreted as commands.
- **Suspicion**: the .cmd batch wrapper is parsing or echoing python output back through the shell. This makes the "official launcher baseline" mandated by the audit protocol unusable on this Windows host.

### [INCONSISTENCY] R21-002 — `providers list` reports only openai while `auth-status` shows codex+claude authenticated
- **Probe**: C
- **Evidence**: `providers list` → `Active: openai`, `Available: openai`. `providers auth-status` → codex OK / claude OK / openai missing.
- **Suspicion**: providers list reads from a config file (only openai configured), while auth-status reads installed CLI accounts. The two views don't reconcile, so codex/claude are authenticated but unselectable.

### [INCONSISTENCY] R21-004 — `surface list` claims Claude Code is "Not connected" while the audit is running through it
- **Probe**: C
- **Evidence**: `surface list` rows: Claude Desktop = Connected; Claude Code = Not connected. The current MCP session is the Claude Code → MCP bridge connection.
- **Suspicion**: `surface list` only checks for the presence of a config file (`~/.claude/...`) at a specific path; the actual connection is detected by file existence, not by live MCP handshake.

### [INCONSISTENCY] R21-007 — `gimo_dashboard` returns `proof_chain_length=0` while a thread-scoped snapshot returns `length=1`
- **Probe**: A
- **Evidence**: `gimo_dashboard` text response includes `proof_chain_length: 0`, immediately after `gimo_get_governance_snapshot(thread_id="r21-probe-a")` returned `proof_chain_length: 1`.
- **Suspicion**: `gimo_dashboard` calls the snapshot helper with no thread_id and gets a global value of zero, instead of reflecting the most-recent or aggregate proof state. Cosmetic but misleading for operators using the dashboard tool as a sanity check.

### [GAP] R21-008 — `model:openai:gpt-4o` is `score=0, policy=blocked` but still the system's active model
- **Probe**: A / C (capabilities and trust state)
- **Evidence**: `/ops/capabilities` → `active_model: gpt-4o, active_provider: openai`. Trust dimension `model:openai:gpt-4o` → `score: 0, policy: blocked, failures: 8`.
- **Suspicion**: the active-model selector is decoupled from the trust engine. A model whose trust policy is `blocked` should either trigger a CascadeService escalation or be removed from `active_model`. Currently the system advertises a blocked model as active, then routes chats into 401 failures (R21-003).

### [FRICTION] R21-009 — `gimo_list_agents` only returns Ollama agents, not the configured providers
- **Probe**: A
- **Evidence**: only `ollama_qwen3-coder_480b-cloud`, `ollama_llama3.2:3b`, `ollama_qwen2.5-coder:3b`. No `openai_gpt-4o`, `codex`, `claude`.
- **Suspicion**: agent registry is populated from auto-discovered Ollama models only; provider-account-backed agents (codex/claude) never get materialized as agents.

### [BLOCKER] R21-010 — Proof chain HTTP/MCP parity violation
- **Probe**: A + D
- **Evidence**: `gimo_verify_proof_chain(thread_id="r21-probe-a")` → `valid:true, length:1`. `GET /ops/threads/r21-probe-a/proofs` → 404 `Thread not found`.
- **Suspicion**: The proof-chain HTTP route requires the thread to exist in `ThreadStore` (the SQLite/JSON thread registry), while `evaluate_action` writes proofs into GICS keyed by an arbitrary `thread_id` string that need not be a registered thread. This violates the surface-parity contract that R20 Change 5 conformance test claimed to enforce — the test only covers the *empty-state* parity (both 404/empty), not the *populated* parity.

### [GAP] R21-011 — `GET /ops/providers` returns 404
- **Probe**: D
- **Evidence**: 404 `Not Found`. CLI providers list works, so the data exists.
- **Suspicion**: providers HTTP router not registered under `/ops/providers`; CLI hits a different path (likely `/ops/providers/list` or via service shim).

### [BLOCKER] R21-012 — `gimo_spawn_subagent provider=ollama_local` fails with `SPAWN_RESOLUTION_FAILED:none:none`
- **Probe**: B
- **Evidence**: Ollama is `CONNECTED` per `gimo_get_status`; trust dimension `model:ollama_local:llama3.2:3b` exists with `score=0.99`; agent `ollama_llama3.2_3b` is in `gimo_list_agents`. Yet the spawn resolver returns `none:none`.
- **Suspicion**: `SubAgentManager._resolve_target_agent` does not match on the provider id `ollama_local` — the agent registry uses an underscored id (`ollama_llama3.2_3b`) but the provider/model tuple resolution path expects a different shape. The structured error is a stub literal, not a real diagnostic.

### [CRITICAL] R21-013 — `auto`/`auto` resolves to NOT-READY provider instead of fallback
- **Probe**: B
- **Tool**: `gimo_spawn_subagent provider=auto model=auto`
- **Expected**: select a ready provider (Ollama is CONNECTED).
- **Actual**: `PROVIDER_NOT_READY:openai:unreachable`.
- **Suspicion**: `auto` resolves by configured-active rather than by readiness — same root cause family as R21-008. CascadeService is not consulted at spawn time.

### [BLOCKED_EXTERNAL] R21-014 — OpenAI / Anthropic / Codex API reachability
- All three configured remote providers report `reachable=false`. No live API key or network path available in this environment. Carried over from R20-010/011.
