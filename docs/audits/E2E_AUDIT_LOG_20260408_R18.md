## 6. Issue List

### #R18-1 [BLOCKER] `gimo_estimate_cost` silently drops `tokens_in` / `tokens_out` and returns hard-coded defaults

- **Probe**: A
- **Tool**: `mcp__gimo__gimo_estimate_cost`
- **Inputs tested**:
  1. `model=gpt-4o-mini, tokens_in=1500, tokens_out=500` → `{input_tokens:1000, output_tokens:500, total_cost_usd:0.00045}`
  2. `model=gpt-5-codex, tokens_in=800, tokens_out=1200` → `{input_tokens:1000, output_tokens:500, total_cost_usd:0.0018}`
- **Expected**: response cost reflects the supplied token counts; response uses canonical `tokens_in` / `tokens_out` keys per R17 Cluster D.
- **Actual**: response always reports `input_tokens:1000, output_tokens:500` regardless of input. Costs scale only with the model price, not the user-supplied volume. Canonical key names are absent from the response.
- **Suspicion**: the MCP tool wrapper for `gimo_estimate_cost` discards the new Pydantic input model and falls through to a default-constructed object before reaching the cost service. R17 Cluster D fixed the inner Pydantic model but the MCP `_register_native` registration still binds to the legacy schema.

### #R18-2 [BLOCKER] MCP schema drift on three native tools (`verify_proof_chain`, `generate_team_config`, plus param-name drift on `propose_structured_plan` / `run_task`)

- **Probe**: A
- **Calls**:
  - `gimo_verify_proof_chain()` with no args → `Required: thread_id`. R17 #11 stated `thread_id: str | None = None`.
  - `gimo_generate_team_config(objective="…")` → `Required: plan_id`. R17 #10 stated `objective` is XOR-accepted.
  - `gimo_run_task(prompt="…")` → `Required: task_instructions`. Param name surfaced to the operator (`prompt`) does not exist; only `task_instructions` is accepted.
  - `gimo_propose_structured_plan(objective="…", repo="…")` → `Required: task_instructions`. Same drift.
- **Expected**: per R17, the new optional / XOR / canonical names are reachable from MCP clients. Per the principle of least surprise, the parameter exposed to the operator matches the schema.
- **Actual**: schemas are stuck on legacy required-only contracts. The R17 fix only landed on the inner Pydantic models, not on the MCP tool registration.
- **Suspicion**: `mcp_bridge/native_tools.py::_register_native` (or its sibling for governance tools) still hand-wires the input shape per tool. The Pydantic models in `native_inputs.py` introduced by R17 Cluster D are not consumed at the registration boundary.

- **Endpoint**: `GET /ops/system/dependencies`
- **Expected**: per R17 Implementation Report §"Cluster C" the response includes `gics_failure_reason` / `message` / `detail` when the daemon failed to start. `gimo_get_gics_insight` should also propagate the same `GicsStartFailure`.

## 9. Architectural Pattern Observed

> **Inner Pydantic / service-layer changes do not propagate to the bridge layer that the operator actually sees.**

The pattern is: **R17 fixed everything one layer deep and called it resolved.** R18 finds the layer above untouched.
