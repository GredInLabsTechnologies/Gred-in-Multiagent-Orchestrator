## Full Issue List

### [BLOCKER] #1 — Run data plane still hollow (PERSISTS from R16 #1, R14 #1)
- **Probe**: A (MCP draft→approve→run)
- **Invocation**: `gimo_create_draft` → `gimo_approve_draft` → `gimo_get_task_status`
- **Expected**: Run transitions through stages, writes `engineering_calc.py`, emits tokens/cost, populates `heartbeat_at`, adds stage log entries
- **Actual**: Run created at 17:16:35.912, "Execution started" log at 17:16:35.933, then **no further activity for ~6 minutes**, finally reports `status:done` with `stage:None`, `heartbeat_at:None`, **3 log entries total**, `commit_after:None`, and **`engineering_calc.py` does not exist on disk**. Metrics: `total_tokens=0, estimated_cost=0.0`.
- **Suspicion**: The draft approval auto-run path wires into a pipeline that never actually dispatches work to the LLM executor / file writer. The fix claimed in R16 ("FileWrite in legacy_run + journal_path injection") either wasn't effective or the relevant code path is not reached when the draft originates from the MCP bridge (vs HTTP direct).

### [BLOCKER] #2 — CLI `chat` and `plan` commands still produce zero stdout (PERSISTS from R16 #2, R14 #2, R13)
- **Probe**: C
- **Invocation**: `python gimo.py chat -m "PONG" -w .`; `python gimo.py plan "Add TODO comment" -w . --no-confirm`
- **Expected**: Renderer prints assistant/orchestrator output; for `plan`, a plan summary
- **Actual**: Both commands hit the 40s timeout with **completely empty stdout**. No error, no spinner, no progress. Exit via timeout.
- **Suspicion**: Either the renderer is discarding SSE frames (ChatRenderer mis-wired), the CLI isn't subscribing to the correct stream, or the underlying chat endpoint is producing no frames because the agentic loop itself is hollow (see #4).

### [BLOCKER] #3 — GICS daemon not initialized (PERSISTS from R16 #3, R14 #3, R13)
- **Probe**: A, D
- **Invocation**: `gimo_get_gics_insight`, `gimo_gics_anomaly_report`, `gimo_get_governance_snapshot`
- **Expected**: `gics_health.daemon_alive:true`, non-empty entries
- **Actual**: All three surfaces report `"GICS not initialized"` / `daemon_alive:false`. R16 fix (token path + pre-check) did not resolve the runtime condition.
- **Suspicion**: The daemon-start path is gated on a precondition that fails silently at boot, or the token-path argument is still ignored by the daemon subprocess, or `GicsService.start_daemon` is mocked in tests but never invoked in the real server startup sequence.

### [BLOCKER] #4 — MCP `gimo_chat` produces hollow orchestrator turn
- **Probe**: A
- **Invocation**: `gimo_chat(message="List files in workspace root...")`
- **Expected**: After background dispatch, polling `/ops/threads/thread_5fd5d6fa` shows an orchestrator turn with non-empty content (tool output or text response)
- **Actual**: Thread ends with `turns:[user, orchestrator]` where **both turns have empty content**. The fire-and-return wrapper returns successfully, but the agentic loop appears to append an empty orchestrator turn and stop.
- **Suspicion**: Same root cause as #1 — the LLM executor in the loop path isn't producing content (possibly provider not wired, model endpoint not called, response stripped). Structurally the tool contract is honored; semantically the loop is dead.

### [CRITICAL] #5 — Runs never emit heartbeat or stage transitions during execution
- **Probe**: A, D
- **Invocation**: poll `/ops/runs/{id}` during a live run
- **Expected**: `heartbeat_at` updates periodically; `stage` cycles through pipeline stages (policy_gate → risk_gate → llm_execute → file_write → ...); `log` accumulates per-stage entries
- **Actual**: `heartbeat_at:null` throughout entire 6+ minute lifetime. `stage:null` throughout. `log` has exactly 2 entries at run creation, then exactly 1 (possibly terminal) on transition to `done`. No pipeline stages observed.
- **Suspicion**: The pipeline runner that should be emitting heartbeat + stage is not running (worker thread not started, or exiting immediately after creating the run row). The `gimo_get_server_info` earlier reported `Worker: not started` for the bridge — may be the same worker or a symptom.

### [CRITICAL] #6 — Zero cost/token tracking across all surfaces (PERSISTS from R16 #6)
