## Full Issue List

### [BLOCKER] #1 — Run Pipeline Hollow Completion (Changed from R14 #1)

- **Probe**: A, B
- **Command/Tool**: `gimo_approve_draft(draft_id, auto_run=true)` → `gimo_get_task_status(run_id)`
- **Expected**: Run executes LLM calls, creates files, records tokens/cost, transitions through stages
- **Actual**: Run transitions `running → done` in ~68s with `stage:null`, `heartbeat_at:null`, `tokens_total:0`, `commit_before/after:null`. SSE events show only 3 log messages. No files created. No tokens consumed.
- **Suspicion**: R15's timeout-wrap causes the pipeline to exit gracefully on timeout instead of hanging. The provider call either never initiates, or times out silently and the pipeline marks itself as "completed" without detecting the failure.

### [BLOCKER] #2 — CLI plan and chat Commands Produce Zero Output (Persists from R14 #2, R13)

- **Probe**: C
- **Command/Tool**: `gimo plan "..." -w <path> --no-confirm`, `gimo chat -m "..." -w <path>`
- **Expected**: Streaming output from LLM generation
- **Actual**: Zero bytes to stdout/stderr within 30 seconds. Process hangs silently. Exit code 124 (timeout).
- **Suspicion**: CLI calls SSE endpoint which calls provider. The provider call never returns (timeout or hang), and the CLI has no progress indicator or timeout handling of its own.

### [BLOCKER] #3 — GICS Daemon Not Initialized (Persists from R14 #3, R13)

- **Probe**: A, B
- **Command/Tool**: `gimo_get_gics_insight`, `gimo_gics_anomaly_report`, `gimo_gics_model_reliability`
- **Expected**: GICS daemon alive, collecting telemetry
- **Actual**: `daemon_alive: false`, `error: "GICS not initialized"` across all GICS tools
- **Suspicion**: Node.js GICS daemon does not start. R15 added a warning log but didn't fix the root cause. May require Node.js >= 18 or the GICS binary may not exist at the expected path.

### [CRITICAL] #4 — gimo_chat MCP Tool Still Times Out (Changed from R14 #5)

- **Probe**: A
- **Command/Tool**: `gimo_chat(message="Hello...", workspace_root="C:/...")`
- **Expected**: Chat response from agentic loop
- **Actual**: `MCP error -32001: Request timed out`
- **Suspicion**: Chat uses a different code path than draft creation. The agentic loop involves multi-turn LLM calls with tool execution, which likely hangs at the provider call. R15's timeout-wrap may not cover the chat path, or the chat timeout is longer than MCP's timeout.

### [CRITICAL] #5 — Trust Score Regression: 0.0 in MCP, 0.85 in HTTP (Worsened from R14 #9)

- **Probe**: A, D
- **Command/Tool**: `gimo_get_trust_profile` vs `POST /ops/trust/query` vs `gimo_evaluate_action`
- **Expected**: Consistent trust score (0.85 default) across all surfaces
- **Actual**:
  - MCP `get_trust_profile`: `{provider:0.0, model:0.0, tool:0.0}`
  - MCP `governance_snapshot`: `{provider:0.0, model:0.0, tool:0.0}`
  - MCP `evaluate_action(write_file)`: `trust_score: 0.0`
  - MCP `evaluate_action(INVALID_TOOL)`: `trust_score: 0.85`
  - HTTP `trust/query`: `score: 0.85, effective_score: 0.85`
- **Suspicion**: R15 Change 3 set `_empty_record::score = 0.85` in trust_engine.py and trust.py, but the MCP bridge calls a different code path that doesn't use `_empty_record`. The HTTP endpoint hits trust.py directly while MCP goes through sagp_gateway.py which may have been broken by R15's removal of the `if score > 0.0 else 0.85` conditional.

### [CRITICAL] #6 — Zero Cost/Token Tracking Across All Operations (Persists from R14 #7)

- **Expected**: Token counts and cost accumulation from LLM operations

### [GAP] #7 — gimo_run_task Reports "Draft creation failed" on Success

### Assessment

**R16 marks the first reduction in total issue count** (11 vs 15 in R14) since R7. Six R14 issues were confirmed fixed, demonstrating real progress in the control plane.

However, the core problem has **morphed, not resolved**:

1. **R6-R14**: Runs stuck at `running` forever → visible failure
2. **R16**: Runs complete to `done` with zero output → **invisible failure**

This is architecturally worse. A stuck run triggers alerts and is debuggable. A hollow completion passes all health checks (`stuck_runs: 0, completion_ratio: 0.9`), giving a false sense of health while producing zero value.

**The root cause remains singular**: the execution path from governance approval through the provider adapter to actual LLM invocation is broken. Everything around it works — governance, policy, trust (partially), cost estimation, threading, observability, CLI, HTTP, MCP — but the data plane (actual work) produces nothing.
