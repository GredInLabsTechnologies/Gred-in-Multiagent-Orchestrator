## NEW Issues Found

### [BLOCKER] #N1 — `run` approves plan but never starts execution
- **Command**: `python gimo.py run d_1775234755369_225011 --no-confirm --timeout 120`
- **Expected**: Plan is approved AND execution begins (tasks are dispatched to LLM workers)
- **Actual**: First attempt: **completely silent** (zero output, exit code 0). Second attempt with `--json`: returns `{"approved": {...}, "run": null}`. Plan was approved but no run was created.
- **Severity**: critical
- **Suspicion**: The `--auto` flag (default) should spawn a backend run after approval, but `"run": null` indicates the run initiation step failed silently. No error message, no indication of what went wrong. The workspace `gimo_prueba` has zero files after "approval".

### [BLOCKER] #N2 — Chat completely unusable in non-interactive environments
- **Command**: `echo "message" | python gimo.py chat`
- **Expected**: At minimum, graceful error. Ideally, single-turn mode for piped input.
- **Actual**: Full traceback crash: `NoConsoleScreenBufferError: Found xterm-256color, while expecting a Windows console`. No fallback to stdin readline, no `--message` flag, no non-interactive mode.
- **Severity**: critical
- **Suspicion**: `prompt_toolkit.PromptSession` is unconditionally instantiated. No `sys.stdin.isatty()` check. No `--message` or `--one-shot` CLI flag.

### [GAP] #N3 — Thread creation ignores user-provided title
- **Command**: `POST /ops/threads?workspace_root=gimo_prueba` with body `{"title": "Calculator Build"}`
- **Expected**: Thread created with title "Calculator Build"
- **Actual**: Thread created with title "New Conversation" — user-provided title silently ignored.
- **Severity**: medium
- **Suspicion**: Thread creation endpoint doesn't read `title` from request body, or the default overwrites it.

### [GAP] #N4 — Agentic loop doesn't parse text-based tool calls from small models
- **Command**: `POST /ops/threads/thread_6977f9a8/chat` with prompt requesting tool use
- **Expected**: LLM tool call is parsed and executed, file is created
- **Actual**: LLM returns tool call as JSON-formatted **text** in `response` field. `tool_calls: []` is empty. The agentic loop does not attempt to parse tool-call-like JSON from the response text. File was never created.
- **Severity**: critical
- **Suspicion**: The OpenAI-compat layer relies on the model using proper tool_call API format. qwen2.5-coder:3b (via ollama) appears to output tool calls as text instead of structured tool_call objects. No fallback parser exists for this case.

### [GAP] #N5 — Plan quality is comically bad for simple tasks
- **Command**: `python gimo.py plan "Build a Python calculator with add, subtract, multiply, divide functions" -w gimo_prueba --no-confirm`
- **Expected**: 2-3 tasks: write code, write tests, verify
- **Actual**: **13 workers** generated including: "Deploy to production environment", "Monitor and maintain in production", "Create user manual", "Package as executable", "Conduct code review with other developers". For a 20-line calculator.
- **Severity**: high
- **Suspicion**: The plan generation prompt does not constrain task count or scope. The small model (3B) lacks judgment to scope appropriately. No plan validation/pruning on the backend.

### [GAP] #N6 — `run` with `--auto` produces zero output in non-JSON mode
- **Command**: `python gimo.py run d_1775234755369_225011 --no-confirm --timeout 120` (no `--json`)
- **Expected**: Some indication of what happened — approval status, run status, progress
- **Actual**: **Completely blank** — exit code 0, zero bytes of output
- **Severity**: high
- **Suspicion**: The `run` command's non-JSON render path doesn't output anything when `run` is null.

### [FRICTION] #N7 — `repos list` dumps raw Python dicts instead of table
- **Command**: `python gimo.py repos list`
- **Expected**: Formatted table like `threads list` or `providers models`
- **Actual**: Raw Python dict output: `{'name': 'foo', 'path': 'C:\\...'}` for each repo, one per line. Inconsistent with every other list command which uses Rich tables.
- **Severity**: medium
- **Suspicion**: `repos list` CLI command prints raw API response without Rich formatting. Missing table renderer.

### [FRICTION] #N8 — `repos list` exposes full filesystem paths with redacted username
- **Command**: `python gimo.py repos list`
- **Expected**: Clean repo names, or at minimum consistent path display
- **Actual**: Paths shown as `C:\\Users\[USER]\\Documents\\Github\\...` — username redacted with `[USER]` but full path exposed. Also shows pytest temp dirs (`pytest-of-shilo\\pytest-524\\test_context_request_active_an0\\dummy_repo`).
- **Severity**: low
- **Suspicion**: Backend scans all git repos under home directory without filtering. Username redaction is cosmetic but leaks temp/test repos.

### [INCONSISTENCY] #N9 — `providers test` says "healthy" for unauthenticated provider
- **Command**: `python gimo.py providers test claude-account`
- **Expected**: Test fails or warns that provider is not authenticated
- **Actual**: "Provider 'claude-account' is healthy." But `providers auth-status` shows claude as "[X] not connected". Contradictory information.
- **Severity**: medium
- **Suspicion**: `providers test` checks if the provider endpoint is reachable (health check) but not if authentication is valid. These are different things but the "healthy" wording implies the provider is ready to use.

### [INCONSISTENCY] #N10 — `providers set` succeeds for unauthenticated provider without warning
- **Command**: `python gimo.py providers set claude-account`
- **Expected**: Warning that provider is not authenticated, or at minimum a hint to authenticate
- **Actual**: `[OK] Active provider set to: claude-account` — pure success message. But any actual use will fail because claude is not authenticated. User discovers failure only when trying to plan/chat.
- **Severity**: medium
- **Suspicion**: `providers set` only updates server-side active provider config. No validation of whether the provider can actually be used.

### [COGNITIVE_LOAD] #N11 — Chat API field naming is non-obvious
- **Command**: `POST /ops/threads/{id}/chat` with `{"message": "..."}`
- **Expected**: Common field name like `message` works
- **Actual**: 422 error — field must be called `content`, not `message`. No helpful error message suggesting the correct field name.
- **Severity**: low
- **Suspicion**: Pydantic model uses `content` field. Standard API design usually uses `message` or `text`. The error shows `"type":"missing","loc":["body","content"]` which does hint at the field name but requires reading JSON carefully.

### [SILENT_FAILURE] #N12 — `run` exits 0 with no output when execution fails to start
- **Command**: `python gimo.py run d_1775234755369_225011 --no-confirm`
- **Expected**: Non-zero exit code or error message when run doesn't start
- **Actual**: Exit code 0, zero output. User has no idea whether the plan was approved, whether execution started, or whether anything happened at all.
- **Severity**: critical
- **Suspicion**: CLI command doesn't validate the `run` field in the response. If `run` is null, it should error or warn.

## PHASE 1 SUMMARY

### Status: BLOCKED at calculator creation step
### Calculator app: NOT BUILT — gimo_prueba/ remains empty

### Total issues: 12 new + 8 confirmed from prior audit = 20 active issues
- **Blockers**: 4 (N1 run-no-execute, N2 chat-crash, prior#2 bond-broken, prior#5 chat-nonTTY)
- **Gaps**: 4 (N3 title-ignored, N4 tool-parse, N5 plan-quality, N6 run-no-output)
- **Errors**: 1 (prior#11 audit-deps-500)
- **Frictions**: 2 (N7 repos-raw-dict, N8 repos-leaks-paths)
- **Inconsistencies**: 3 (N9 test-vs-auth, N10 set-no-warn, prior#19 config-mismatch)
- **Cognitive Load**: 1 (N11 field-naming)
- **Silent Failures**: 1 (N12 run-exit-0)

### Critical path blockers (why calculator was not built):
1. **Chat is unusable** — crashes in non-TTY (BLOCKER #5/N2), only way to do agentic work
2. **Run doesn't execute** — plan was generated and approved but `run` never dispatched tasks (N1)
3. **Tools don't fire** — even via direct API, agentic loop returns tool calls as text, never executes them (N4)
4. **Plan quality is unusable** — 13 absurd tasks for a 20-line app, including "deploy to production" (N5)

### What the prior audit's fixes actually solved:
The 3 confirmed fixes (`gimo up`, `providers set` auth, `mastery status`) are real improvements in **infrastructure** — server lifecycle and provider management work now. The 4 partial fixes (empty-state messages) improve **polish**.

**But the core product loop — plan → run → tools → code — remains completely broken.** A user still cannot get GIMO to write a single line of code through any CLI pathway.

### Suspicions for Phase 2:
1. **Run dispatch**: The approve→run pipeline has a broken handoff. `run` returns null but no error.
2. **Tool calling**: The OpenAI-compat layer doesn't inject tools schema into the LLM call, or ollama's API doesn't support tool_call response format for this model.
3. **Plan generation**: No constraints on task count. LLM hallucinates absurd scope because the system prompt doesn't say "keep it minimal".
4. **Bond system**: Fernet encryption/decryption of tokens is broken between CLI invocations — likely machine key derivation issue.
5. **Chat non-TTY**: Fix is simple (`sys.stdin.isatty()` check + readline fallback) but has never been implemented.
