## Full Issue List

### [BLOCKER] #1 — Run Execution Never Progresses (Persists from R13)

- **Probe**: A, B, D
- **Command/Tool**: `POST /ops/drafts/{id}/approve?auto_run=true` → `GET /ops/runs/{id}`
- **Expected**: Run progresses through stages (planning → executing → done), LLM calls made, tokens counted
- **Actual**: Run stays at `status:running, stage:null, heartbeat_at:null` with `tokens_total:0` for 10+ minutes. Worker reported as "not started" in server_info. After `gimo_reload_worker`, still no progress.
- **Suspicion**: RunWorker spawns via `asyncio.create_task()` but the execution coroutine either exits silently, is waiting for a condition never met, or the provider connection path is broken.

### [BLOCKER] #2 — CLI plan and chat Commands Produce Zero Output (Persists from R13)

- **Probe**: C
- **Command/Tool**: `gimo plan "..." -w <path> --no-confirm`, `gimo chat -m "..." -w <path>`
- **Expected**: Streaming output from LLM generation
- **Actual**: Zero bytes to stdout/stderr within 60 seconds. Process hangs silently.
- **Suspicion**: CLI calls SSE endpoint which calls provider, but the provider call never completes. R13 changed to stdin piping, but the provider subprocess may hang waiting for input or the SSE connection may never emit.

### [BLOCKER] #3 — GICS Daemon Not Initialized (Persists from R13)

- **Probe**: A, B
- **Command/Tool**: `gimo_get_gics_insight`, `gimo_gics_anomaly_report`, `gimo_gics_model_reliability`
- **Expected**: GICS daemon alive, collecting telemetry
- **Actual**: `daemon_alive: false`, `error: "GICS not initialized"` across all GICS tools
- **Suspicion**: R13 fixed `config.py` path to `cli/index.js` and added immediate health ping, but daemon still not starting. Either Node.js/GICS binary not available, or startup still fails silently.

### [BLOCKER] #4 — All LLM-Dependent MCP Tools Timeout

- **Suspicion**: These tools call LLM providers internally. The provider call hangs (same root cause as #1 and #2), exceeding the MCP stdio timeout (default 30-60s).

### Assessment

R14 shows the **control plane is stable and maturing** — all read-only governance, observability, and informational tools work correctly. R13 fixes for the MCP bridge crash (#1, #2), auth/check (#5), SSE events (#8), /ops/openapi (#9), budget data (#13), and mastery analytics (#14) are all holding.

However, the **data plane remains completely non-functional**:
1. **No LLM execution occurs** — runs stay at stage:null forever
2. **CLI agentic commands (plan, chat) produce zero output** — same as R13
3. **GICS daemon refuses to start** — reported fixed in R13 but still dead
4. **All LLM-dependent MCP tools timeout** — same pattern

The core issue is singular: **the execution path from governance approval to actual LLM provider invocation is broken**. Every downstream problem (zero cost tracking, empty trust, empty GICS, CLI silence) traces back to this one failure. Until a run can actually execute an LLM call and complete, the governance shell — which is well-built and functional — has nothing to govern.

New issues in R14 not seen in R13:
- **#5**: `gimo_chat` MCP tool sends malformed HTTP body (422)
- **#6**: `gimo_spawn_subagent` requires `workspace_path` not exposed in MCP schema
- **#8**: `gimo_generate_team_config` cannot find drafts as plans
- **#12**: `POST /ops/threads` requires undocumented query param

These represent MCP bridge serialization gaps introduced or exposed by continued testing.
