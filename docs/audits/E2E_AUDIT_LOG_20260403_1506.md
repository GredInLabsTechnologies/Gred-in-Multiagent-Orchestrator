## Issue Log

### [BLOCKER] #1 — `gimo up` silently fails to start the server
- **Command**: `python gimo.py up`
- **Expected**: Server starts and is reachable at http://127.0.0.1:9325
- **Actual**: Command outputs "Starting GIMO server on http://127.0.0.1:9325..." but server never becomes reachable. `gimo ps` returns "No GIMO instances found". Had to use `python -m uvicorn tools.gimo_server.main:app --port 9325` directly.
- **Severity**: critical
- **Suspicion**: Background process spawning fails silently on Windows — no PID tracking, no health check, no error output.

### [BLOCKER] #2 — Bond system broken: login succeeds → next command says "expired"
- **Command**: `gimo login` → success → `gimo providers set claude-account` → "Bond expired or invalid"
- **Expected**: After successful login, subsequent commands should use the bond
- **Actual**: Login creates bond file at `~/.gimo/bonds/b0665e6f54cb06c7.yaml` with encrypted token, but EVERY subsequent command shows "Bond expired or invalid. Run: gimo login". Commands still work via legacy fallback, making the warning confusing.
- **Severity**: critical
- **Suspicion**: Token encryption/decryption cycle is broken — token_encrypted is stored but cannot be decrypted on subsequent reads. Fernet key derivation likely machine-specific and fails between invocations.

### [BLOCKER] #3 — `providers set` requires admin role, CLI login only grants operator
- **Command**: `gimo providers set claude-account --model claude-sonnet-4-6`
- **Expected**: Operator role should be able to set their own provider
- **Actual**: CLI error "[X] Failed to set provider (401): Invalid token". Direct API call reveals: `"admin role or higher required"`. The CLI `gimo login` creates a bond with `role: operator`, which lacks permission to change providers.
- **Severity**: critical
- **Suspicion**: `/ops/provider/select` endpoint has overly restrictive role gate. There is no CLI path to get admin role — only `.env` `ORCH_TOKEN` grants admin via direct API.

### [BLOCKER] #4 — `providers login claude` requires interactive browser — no headless/API-key flow
- **Command**: `python gimo.py providers login claude`
- **Expected**: Some way to authenticate with Claude provider (API key, env var, or device code)
- **Actual**: Opens a browser tab for OAuth. Response: `{'status': 'pending', 'message': 'Se ha abierto una pestaña en tu navegador...'}`. No API key input option. No env var fallback. No `--api-key` flag.
- **Severity**: critical
- **Suspicion**: Only OAuth device flow implemented for Claude provider. Missing: `--api-key` flag, `ANTHROPIC_API_KEY` env var detection, or `gimo providers set claude-account --api-key <key>` path.

### [BLOCKER] #5 — `chat` crashes in non-TTY environments
- **Command**: `echo "Create calculator" | python gimo.py chat`
- **Expected**: Chat accepts piped input or at least fails gracefully
- **Actual**: Full traceback: `NoConsoleScreenBufferError: Found xterm-256color, while expecting a Windows console`. Crash in prompt_toolkit's Win32Output → get_win32_screen_buffer_info.
- **Severity**: critical
- **Suspicion**: `PromptSession` from prompt_toolkit requires a real Windows console. No fallback for piped/redirected stdin. Chat is unusable from any automated/scripted environment.

### [BLOCKER] #6 — `plan` command fails with 404 when calling ollama through server
- **Command**: `gimo plan "Build a Python calculator" -w gimo_prueba --no-confirm`
- **Expected**: Plan generated using local ollama provider
- **Actual**: Error: "Plan generation failed: Client error '404 Not Found' for url 'http://localhost:11434/v1/chat/completions'". But direct curl to the same URL works perfectly.
- **Severity**: critical
- **Suspicion**: GIMO server's OpenAI-compat client is not sending the request correctly through its proxy layer. Model ID mismatch or incorrect path routing inside the server.

### [GAP] #7 — Provider sessions don't persist between CLI sessions
- **Command**: Configure provider → restart CLI → provider config lost
- **Expected**: Once a provider is configured and authenticated, the session should persist
- **Actual**: Provider state must be re-established each session. No durable credential storage for LLM providers.
- **Severity**: high
- **Suspicion**: Provider credentials stored only in server runtime memory, not persisted to disk. Server restart wipes auth state.

### [GAP] #8 — No CLI command to create/register a new workspace/repo
- **Command**: `gimo repos --help` → only `list` and `select`
- **Expected**: `gimo repos add <path>` or `gimo repos create <name>` to register a new project
- **Actual**: Only `list` (shows all Github dirs on disk) and `select` (marked as "Legacy host-path selection is not part of the canonical terminal flow"). No way to register `gimo_prueba` as a workspace through CLI.
- **Severity**: medium
- **Suspicion**: Repo management designed for UI, not CLI. CLI assumes workspace = cwd.

### [GAP] #9 — Agentic chat doesn't execute tools — only describes what to do
- **Command**: (from previous thread) "Use the write_file tool to create calculator.py"
- **Expected**: LLM uses tool-calling to create the file
- **Actual**: LLM responds with text describing steps: "To use the write_file tool... you can follow these steps: 1. Use the write_file tool..." — never actually calls tools. Across 8 turns, zero tool executions.
- **Severity**: critical (upgraded from GAP — this defeats the entire product purpose)
- **Suspicion**: System prompt for agentic loop may not be sending tools schema correctly to the LLM, or the local model (qwen2.5-coder:3b) is too small to reliably do tool-calling. No client-side enforcement of tool-use format.

### [ERROR] #10 — `mastery status` returns 500 "protective defense mode"
- **Command**: `gimo mastery status`
- **Expected**: Token economy status display
- **Actual**: `Failed (500): {'error': 'Internal System Failure', 'correlation_id': '...', 'threat_level': 'NOMINAL', 'message': 'System has entered protective defense mode.'}`
- **Severity**: high
- **Suspicion**: Missing service dependency or uninitialized state triggers a global error handler that masks the real error behind a dramatic "defense mode" message.

### [ERROR] #11 — `audit` → Dependencies 500, Audit Tail 403
- **Command**: `gimo audit`
- **Expected**: Clean audit summary
- **Actual**: Table shows `Dependencies: 500, n/a dependencies` and `Audit Tail: 403, no audit lines`
- **Severity**: medium
- **Suspicion**: Dependencies endpoint has a bug (500), audit tail endpoint requires higher permissions than operator.

### [FRICTION] #12 — Every command shows "Bond expired or invalid" warning
- **Command**: Any command (status, providers list, threads list, etc.)
- **Expected**: Commands work silently or show bond status only when relevant
- **Actual**: EVERY single command prints "Bond expired or invalid. Run: gimo login" as first line, even though commands still work (via legacy fallback). Trains users to ignore the warning.
- **Severity**: medium
- **Suspicion**: Bond validation runs unconditionally before command dispatch. Since bond decryption is broken (#2), warning fires always. Legacy token fallback silently saves the request.

### [FRICTION] #13 — `trust status` shows empty table with no explanation

### [COGNITIVE_LOAD] #18 — Login requires manual token discovery
- **Command**: `gimo login` (interactive) or via env var

- **Actual**: Prompt says "Enter server token (from server's .gimo_credentials or ORCH_OPERATOR_TOKEN):" but `.gimo_credentials` doesn't exist. Actual token is in `.orch_token`. Non-interactive mode requires `ORCH_OPERATOR_TOKEN` env var. No `--token` flag.
- **Severity**: medium
- **Suspicion**: Help text references a file that doesn't exist or has been renamed. Missing `--token <file>` convenience flag.

### [INCONSISTENCY] #19 — config shows `preferred_model: claude-haiku-4-5-20251001` but active provider uses `qwen2.5-coder:3b`
- **Command**: `gimo config --show` vs `gimo status`

- **Actual**: Config says `preferred_model: claude-haiku-4-5-20251001`, status says `Provider: local_ollama / qwen2.5-coder:3b`. Two sources of truth.
- **Severity**: medium
- **Suspicion**: `config.yaml` preferred_model is ignored when backend has a different active provider. No sync between local config and backend state.

### [INCONSISTENCY] #20 — `repos select` marked "Legacy" but no replacement exists

- **Actual**: Help says "Legacy host-path selection is not part of the canonical terminal flow." but no canonical alternative is documented or available.
- **Severity**: low

### [SILENT_FAILURE] #21 — `gimo up` returns success-looking output but server never starts
- **Command**: `gimo up`
- **Expected**: Clear success/failure indication
- **Actual**: "Starting GIMO server on http://127.0.0.1:9325..." — looks like it's working. No error, no "failed", no timeout. Server never starts. `gimo ps` confirms nothing running.
- **Severity**: critical (duplicate of #1, different angle)
- **Suspicion**: Fire-and-forget subprocess spawn with no health verification. Exit code 0 even when server fails.
