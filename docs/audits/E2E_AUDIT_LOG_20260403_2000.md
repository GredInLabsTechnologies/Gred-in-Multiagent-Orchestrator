## NEW Issues Found

### [BLOCKER] #N1 — Agentic chat tools denied by `propose_only` policy, then `propose_plan` fails schema validation

- **Command**: `curl -X POST /ops/threads/{id}/chat` with content requesting file creation
- **Expected**: LLM calls `write_file` tool, file is created in workspace
- **Actual**: Three sequential failures:
  1. `write_file` → `policy_denied`: "Tool 'write_file' not in allowed_tools for policy 'propose_only'"
  2. `propose_plan` → `error`: "Task 0 missing 'id' or 'title'"
  3. `propose_plan` → `error`: "Task task_1 missing 'agent_rationale'"
  Response says "Plan proposed. Please review and approve to continue." but NO plan was actually proposed.
- **Severity**: critical
- **Suspicion**: Default thread agent preset is `plan_orchestrator` with `propose_only` execution policy. This policy blocks all file mutation tools. The LLM then falls back to `propose_plan` but can't satisfy the strict schema validation (requires `agent_rationale`, specific field structure). The response claims success despite total failure.

### [BLOCKER] #N2 — Windows cp1252 encoding crash on ALL Rich output containing Unicode

- **Command**: `python gimo.py chat -m "What files are in this project?"` AND `python gimo.py threads show thread_xxx`
- **Expected**: Response rendered to terminal
- **Actual**: `UnicodeEncodeError: 'charmap' codec can't encode character '\U0001f527' in position 4` (emoji 🔧) or `'\u25b8'` (▸ triangle). Crashes at `rich/_windows_renderer.py:19` → `cp1252.py:19`.
- **Severity**: critical
- **Suspicion**: Rich's legacy Windows renderer uses `cp1252` encoding. Any LLM response containing emoji or Unicode symbols (▸, 🔧, etc.) crashes the entire CLI. This is systemic — affects `chat -m`, `threads show`, and likely any command that renders LLM-generated content. The piped/non-TTY `echo | gimo chat` path triggers a DIFFERENT crash (also Rich, but via tool_call_start rendering).

### [GAP] #N3 — `providers login` accepts any string as valid API key without validation

- **Command**: `python gimo.py providers login claude --api-key "invalid-key-test"`
- **Expected**: Key is validated against the provider API before being stored, or at minimum a warning
- **Actual**: `[OK] claude authenticated and set as active provider`. `auth-status` shows `[OK] authenticated / api_key`. The key `"invalid-key-test"` is stored and will cause opaque failures later when actually trying to use the provider.
- **Severity**: high
- **Suspicion**: `providers login --api-key` stores the key server-side without making a test API call. No key format validation (Anthropic keys start with `sk-ant-`).

### [GAP] #N4 — No way to create a new repo through CLI

- **Command**: `python gimo.py repos --help`
- **Expected**: `repos create` or `repos register` subcommand
- **Actual**: Only `list` and `select` (deprecated). To work with a new repo, user must navigate to the directory and run `gimo init`, which initializes `.gimo/` but doesn't register the repo with the server in a way that makes it selectable.
- **Severity**: medium
- **Suspicion**: Repo registration happens implicitly through workspace detection, not through explicit CLI action. The `repos list` shows repos discovered by scanning the filesystem, not repos the user has intentionally registered.

### [GAP] #N5 — `run` never creates actual execution — plan→run pipeline broken

- **Command**: `python gimo.py run d_1775244283787_6b5ac9 --no-confirm`
- **Expected**: Plan is approved AND backend run is created with tasks dispatched
- **Actual**: Exit 1: "Draft approved but run was NOT started. Possible causes: Plan was not eligible for auto-run (missing execution_decision)". The error message is helpful (improvement!) but the root problem remains: no plan is ever "eligible for auto-run".
- **Severity**: critical
- **Suspicion**: The backend requires an `execution_decision` field or similar gate that structured plans don't include. Every plan→run attempt fails at the same handoff point.

### [FRICTION] #N6 — Ollama-generated plans have generic placeholder titles

- **Command**: `python gimo.py plan "Build hello world in Python" -w gimo_prueba --no-confirm --json` (with local_ollama active)
- **Expected**: Worker titles describe what the worker actually does (e.g., "Write hello_world.py")
- **Actual**: Worker title is "[WORKER] Execute specific task" with description "Perform actual work". No mention of hello world, Python, or the actual objective.
- **Severity**: medium
- **Suspicion**: The small model (qwen2.5-coder:3b) doesn't have enough context or capability to generate meaningful task descriptions. The system prompt templates may not constrain output quality sufficiently for small models.

### [FRICTION] #N7 — `plan ""` blocks on interactive confirmation instead of failing fast

- **Command**: `python gimo.py plan ""`
- **Expected**: Immediate validation error (empty description)
- **Actual**: First shows "Workspace: ..." then prompts "Proceed? [Y/n]:" — only after the interactive prompt does it abort. With `--no-confirm --json` it correctly returns 422. But without flags, the confirmation prompt fires before input validation.
- **Severity**: low
- **Suspicion**: Workspace confirmation runs before input validation in the CLI command flow.

### [INCONSISTENCY] #N8 — `providers test` reports "Auth status: unknown" even after successful login

- **Command**: `python gimo.py providers login claude --api-key "..."` → `python gimo.py providers test claude-account`
- **Expected**: "Auth status: authenticated" (consistent with `auth-status` which shows `[OK] authenticated`)
- **Actual**: `providers test` says "Auth status: unknown". `providers auth-status` says "[OK] authenticated". Two commands give contradictory information about the same provider.
- **Severity**: medium
- **Suspicion**: `providers test` checks endpoint reachability but doesn't query the auth subsystem. The auth-status endpoint and the test endpoint use different code paths.

### [SILENT_FAILURE] #N9 — Chat API response claims "Plan proposed" when no plan was actually created

- **Command**: `curl -X POST /ops/threads/{id}/chat` (see N1 above)
- **Expected**: If plan proposal fails validation, response should indicate failure
- **Actual**: `"status":"ok"`, `"response":"Plan proposed. Please review and approve to continue."`, `"finish_reason":"plan_proposed"`. But examining the `tool_calls` array shows both `propose_plan` calls returned `"status":"error"`. The top-level response contradicts the tool-level reality.
- **Severity**: high
- **Suspicion**: The agentic loop uses the LLM's text response as the top-level response, ignoring tool execution failures. The LLM says "Plan proposed" because it thinks it did, but the tool execution disagrees.

### [COGNITIVE_LOAD] #N10 — `config --model` doesn't affect plan generation provider

- **Command**: `python gimo.py config --model claude-haiku-4-5-20251001` → `python gimo.py plan "..." --json`
- **Expected**: Plan generated using haiku model
- **Actual**: Plan generated with active provider's model (claude-sonnet-4-6). Config `preferred_model` and `providers set` are independent systems with unclear precedence.
- **Severity**: low
- **Suspicion**: `config.preferred_model` is stored locally but the backend uses the active provider's model. Two model selection systems (config vs providers) with no clear documentation of which takes precedence.

### [FRICTION] #N11 — `watch` produces no output and silently times out

- **Command**: `python gimo.py watch --limit 3 --timeout 10 --json`
- **Expected**: Either events or a "no events" message
- **Actual**: Runs for 10 seconds, produces empty output (single newline), exits 0
- **Severity**: low
- **Suspicion**: SSE stream has no events to send. No timeout message, no "waiting for events..." indicator, no "0 events received" summary.

3. **Windows encoding**: Even when chat `-m` gets a response, it crashes rendering it. Any Unicode character outside cp1252 range (emoji, special symbols) kills the CLI.

### Suspicions for Phase 2:

1. **Policy architecture**: The `propose_only` execution policy is the default for all new threads. There's no CLI-accessible way to create a thread with a different policy (e.g., `execute` or `autonomous`). The policy system may be designed for the web UI approval flow, not CLI usage.

2. **`propose_plan` schema strictness**: The tool schema requires `agent_rationale` per task, but the LLM doesn't know this. The tool definition given to the LLM may not include all required fields, or the validation is stricter than the schema advertises.

3. **Run eligibility**: The `execution_decision` gate that prevents runs from starting may require a field that the plan generation endpoint never sets. Circular dependency: plan can't include execution_decision, run requires it.

4. **Windows encoding**: Rich Console on Windows defaults to legacy renderer when terminal doesn't support VT100. The fix is likely `Console(force_terminal=True)` or filtering non-ascii from output, or using `PYTHONIOENCODING=utf-8`.

5. **Chat response honesty**: The agentic loop should surface tool execution failures in the top-level response, not echo the LLM's optimistic text.
